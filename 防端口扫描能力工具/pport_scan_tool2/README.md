# Web 应用与 API 漏洞防护扫描工具

## 1. 工具简介

这是我们搭建的一款面向 **Web 应用与 API 安全防护验证** 的工具环境。它基于机器学习检测引擎，在 HTTP 请求入口处对访问流量进行实时分析，识别并拦截偏离正常用户行为模式的可疑请求，从而帮助我们验证 Web 应用在面对常见攻击和未知攻击时的防护能力。

与传统只依赖固定规则的安全检测方式不同，我们的工具结合了监督模型与无监督模型：一方面可以识别已知的攻击模式，另一方面也能够学习当前业务环境中的正常访问特征，用于发现异常请求和潜在零日攻击行为。

本工具采用独立部署方式，适用于授权测试、教学演示、安全防护能力验证和实验环境下的攻防对照分析。

## 2. 我们的工具能防御什么？

我们的工具部署在 HTTP 请求入口处，每一条请求在到达后端应用之前，都会经过安全引擎的实时检测。整体检测过程主要包含两个阶段：

### 2.1 监督模型检测

监督模型基于大量恶意请求与良性请求样本训练得到，能够识别常见的 Web 攻击模式，例如：

- SQL 注入
- 敏感文件探测
- 代码仓库泄露探测
- 路径穿越
- 命令注入
- 跨站脚本攻击
- 异常扫描请求
- OWASP Top 10 中的常见风险行为

### 2.2 无监督模型检测

无监督模型会学习当前实验环境或业务系统中的正常流量特征。当某个请求明显偏离正常访问模式时，系统会进一步判断该请求是否具有攻击意图。

这种方式可以帮助我们发现一些难以通过固定规则直接覆盖的异常行为，尤其适合用于验证未知攻击、变形攻击和零日攻击场景下的防护能力。

### 2.3 置信度评分与拦截决策

经过两阶段分析后，系统会为请求生成风险置信度评分，并根据当前安全策略决定：

- 放行正常请求；
- 标记可疑请求；
- 直接阻断高风险请求。

在拦截模式下，高风险请求会被安全引擎阻断，通常返回 `403 Forbidden`，避免攻击流量继续进入后端应用。

## 3. 快速开始

### 3.1 环境要求

本实验环境需要满足以下条件：

- Linux 操作系统  
  本实验在 Ubuntu / WSL2 环境下完成。
- Docker
- Docker Compose
- curl  
  用于发送测试请求和验证防护效果。

## 4. 下载与安装

根据原项目说明，在 Linux 环境下可以通过以下命令快速安装：

```bash
wget https://downloads.openappsec.io/open-appsec-install && chmod +x open-appsec-install
./open-appsec-install --auto
```

在本实验中，我们采用 Docker 部署方式，相关服务已经通过 `docker-compose.yaml` 编排完成，便于快速启动、重启和复现实验结果。

## 5. 准备实验环境

首先进入 WSL2 的 Ubuntu 环境：

```bash
wsl -d Ubuntu
```

然后进入实验目录：

```bash
cd ~/openappsec-lab
```

## 6. 启动防护环境

启动整套实验环境，包括核心检测组件、前端 NGINX 组件和后端测试应用：

```bash
docker compose up -d
```

重启核心检测组件：

```bash
docker compose restart appsec-agent
```

重启前端 NGINX 组件，确保请求会经过安全引擎：

```bash
docker compose restart appsec-nginx
```

等待服务稳定运行：

```bash
sleep 30
```

## 7. 应用防护策略

加载本地安全策略，使工具进入拦截模式：

```bash
docker exec appsec-agent open-appsec-ctl --apply-policy /ext/appsec/local_policy.yaml
```

策略应用完成后，外部请求会先进入前端 NGINX，再经过安全引擎检测，最后才可能转发到后端测试应用。

## 8. 对照验证实验

为了验证工具是否真正起到防护作用，我们设计了两组访问路径进行对照：

- **未应用工具：直连后端应用**  
  请求直接到达后端测试应用，不经过安全引擎。
- **应用工具后：经过防护入口访问**  
  请求先进入安全防护入口，再由安全引擎判断是否允许转发。

如果同样的攻击请求在直连后端时可以到达应用，而经过防护入口后被拦截，则说明工具在实际链路中发挥了应用层安全防护作用。

## 9. 实验一：敏感文件探测

### 9.1 未应用工具：直连后端

```bash
docker run --rm --network openappsec-lab_default curlimages/curl:latest -s -D - -o /dev/null "http://juiceshop-backend:3000/.env" | head -n 1
```

### 9.2 应用工具后：经过防护入口

```bash
curl -s -D - -o /dev/null "http://localhost/.env" | head -n 1
```

### 9.3 预期结果

直连后端时，请求可能正常到达后端应用；经过防护入口后，请求应被识别为敏感文件探测行为，并返回类似结果：

```text
HTTP/1.1 403 Forbidden
```

```text
HTTP/1.1 403 Forbidden
```

```text
HTTP/1.1 403 Forbidden
```

## 12. 实验四：搜索参数注入探测

### 12.1 未应用工具：直连后端

docker run --rm --network openappsec-lab_default curlimages/curl:latest -s -D - -o /dev/null "http://juiceshop-backend:3000/rest/products/search?q=%27%20OR%201=1--" | head -n 1

### 12.3 预期结果

预期返回结果如下：

```text
HTTP/1.1 403 Forbidden
```

## 13. 实验结论

通过以上对照实验可以看到，同样的攻击请求在不经过本工具时，可以直接到达后端测试应用；而经过本工具防护入口后，请求会被安全引擎识别并阻断，返回 `403 Forbidden`。

这说明我们的工具已经在实际访问链路中起到了应用层安全防护作用，能够对敏感文件探测、代码仓库泄露探测、SQL 注入和参数注入等典型 Web 攻击行为进行有效拦截。

## 14. 工具特点

### 14.1 独立部署，便于验证

我们的工具可以作为独立防护组件部署在 Web 应用前端，不需要修改后端业务代码，适合用于实验环境、防护能力验证和教学演示。

### 14.2 实时检测，链路清晰

所有外部请求都会先经过安全入口，再由检测引擎进行判断。通过直连后端与经过防护入口的对照实验，可以清楚展示安全组件的实际拦截效果。

### 14.3 结合监督与无监督模型

工具不仅可以识别已知攻击模式，还可以基于当前环境的正常流量特征发现异常请求，从而提升对变形攻击和未知攻击的识别能力。

### 14.4 支持策略配置

通过加载本地安全策略，我们可以灵活控制工具的检测模式、拦截模式和防护规则，便于开展不同类型的实验验证。

## 16. 注意事项

本工具仅用于 **合法授权的安全测试、教学实验和防护能力验证**。请不要将其用于未授权目标的扫描、攻击或绕过行为。

在进行实验时，建议使用本地靶场、测试应用或明确授权的系统，避免对真实业务系统造成影响。
